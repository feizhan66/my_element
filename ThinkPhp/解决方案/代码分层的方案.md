# 代码分层的方案

```
如果你在开发一个商城系统.拥有APP(HTTP API),WAP,以及PC版.那么他们都有共同的业务逻辑.比如都有注册,购物,订单查询功能.
那么service可以提供完整的一个操作,比如删除一个订单.即控制器层与service层交互.
 - service,层调用 财务,会员,订单,产品,信息通知等若干logic层执行相关任务.
 - logic,在去与model交互.完成数据库操作.
如果不这么分层的话.那么订单删除对各个逻辑之间的调用.就要写上N份(APP API\网页版控制器\WAP控制器).
如果不做逻辑层的话.那么那个各个职能模块的定位就回模糊.
如果逻辑层不和Model之间分离的话.那么数据库与职能模块之间的耦合度会变大.
当然我这里能力有限也做不到太大架构.打算先把Model和logic先写在一起.以后在分..
```
